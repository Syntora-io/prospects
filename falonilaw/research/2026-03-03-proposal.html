Understanding Your Situation

Faloni Law processes over 1,000 court emails per day through the NJ E-Courts system. Every email contains data that must be extracted, validated, and loaded into JST CollectMax -- your central database -- as fast and accurately as possible. Your vendors in India and Africa need real-time accurate data. Your filings go out in bulk. The margin for error is zero.

Your programming team -- Sean, Chris, Brian, and Darsh -- builds and maintains the Python scripts and PowerShell endpoints that power this data pipeline. They're good at what they do. The problem is not their skill. The problem is how they work together.

What We Identified

  • Code is siloed. Each developer works on their own AWS Workspace. There is no shared repository, no visibility into what others are building, and no way to recover code if a Workspace fails.
  • No code review process. Scripts go to production without a second pair of eyes. Edge cases, bugs, and misconfigurations are caught after they cause problems, not before.
  • Lead programmers are bottlenecked. Chris and Sean carry the highest-priority work and can't get to backlog items. Adding another programmer into the current setup doesn't solve this -- it adds another silo.
  • No onboarding path. When a new developer starts, Sean walks them through everything manually. There's no documentation, no central codebase to reference, no way to ramp up independently.
  • No audit trail on code changes. There is no record of what changed in a script, when it changed, who changed it, or who approved the change.

Our Approach

We are not proposing to replace JST. JST is your engine -- it handles accounts, letters, payments, and compliance inside the software. DevOps controls everything outside it: the server JST runs on, the scripts that move data in and out, the deployment process, the security model, and the audit trail.

We are not proposing to rewrite your existing scripts. Sean's scrapers work. Darsh's scrapers work. We are not touching what works.

We are proposing one platform -- GitHub -- that solves eight problems at once.

Before and After

Scenario: A developer writes a new email scraper for a court keyword.

Current Workflow

  • Developer writes the script on their Workspace
  • Nobody else can see it
  • No review before it runs in production
  • If the Workspace fails, the code is gone
  • If the developer leaves, the code leaves
  • No record of what changed or when

With GitHub

  • Developer writes the script, pushes to their branch
  • Sean reviews and approves via pull request
  • Automated checks catch errors before merge
  • Code merges to main, deploys automatically
  • Full history is preserved permanently
  • Every change and approval is logged

Production Protection Rules

  • Every change to production requires a pull request -- no direct push
  • At least one approval required (Sean or Trisha depending on scope)
  • Automated checks must pass before code can merge
  • No one can rewrite history or force-push to production
  • Complete log of every change, every approval, every deployment -- permanent and unalterable

Automatic Reviewer Assignment

GitHub supports ownership rules that automatically assign the correct reviewer based on which files a pull request touches. If someone changes a filing script, both Sean and Trisha are notified. If someone changes a scraper, Sean is notified. No one has to remember to tag the right person. The system enforces it.

Implementation Stages

Each stage builds on the previous one. Each stands on its own. You can pause between stages if needed.

Stage 1: GitHub + Documentation

  • Set up GitHub Organization and shared repository
  • Import all existing scripts from each developer's Workspace
  • Document current JST setup, data flows, and script inventory

Stage 2: Automation Pipeline

  • Automated linting and validation on every pull request
  • Auto-deploy from the repository to AWS Workspaces on merge
  • Staging environment for pre-production testing
  • AWS backup configuration and recovery testing
  • Enable secret scanning and dependency vulnerability alerts

Stage 3: ETL Pipeline + Audit Trail

  • MaxExtract and MaxUpdater wrapper scripts in the repository
  • Nightly ETL pipeline: extract account data, validate, load into secure database
  • Append-only audit table logging all data access and pipeline runs
  • Compliance gate: required reviewer on code paths touching debtor communication

Stages 4-5: Future State

  • Advanced security scanning and compliance automation
  • Infrastructure-as-code for new AWS resources

  • Evaluated and implemented together when the foundation is stable
